# 杯中事 PRD

> 文档版本：V1.0  
> 产品阶段：微信小程序 MVP  
> 更新日期：2026-07-17

## 1. 产品概述

### 1.1 产品名称

杯中事

### 1.2 产品定位

杯中事是一款面向鸡尾酒入门用户的微信小程序。它帮助用户在酒吧看懂酒单、根据当晚的口味偏好选择一杯酒，并随手记录喝过的酒。

产品不是鸡尾酒百科，也不以教学调酒为核心。第一版只解决三个问题：选一杯、识一杯、记一杯。

### 1.3 目标用户

- 喜欢喝酒，但对鸡尾酒和基酒了解不多。
- 在酒吧面对陌生酒名和配料时难以判断味道。
- 知道自己大致喜欢什么，却不知道应该点哪一杯。
- 希望留下简单饮用记录，但不需要社交、评分或复杂管理。

### 1.4 核心使用场景

1. 用户在酒吧看不懂酒单，希望快速知道某杯酒或一组配料大致是什么味道。
2. 用户不知道今晚喝什么，希望根据自己的口味得到一个明确推荐。
3. 用户喝到一杯酒后，想简单记下酒名、配料或当时的感受。

### 1.5 产品目标

- 让不了解鸡尾酒的用户在 1 分钟内完成一次选酒。
- 用易懂、可信的语言解释酒款，降低看酒单的门槛。
- 推荐结果只给一杯，并说清楚为什么适合用户。
- 通过简单记录帮助用户逐渐认识自己的口味。

### 1.6 非目标

第一版不包含：

- 调酒教程、精确比例、器具和制作步骤。
- 无酒精鸡尾酒。
- 用户账号、微信登录和云端同步。
- 社交、评论、评分、排行榜和分享社区。
- 拍照识别酒单、图片上传和 AI 自由生成答案。
- 酒吧推荐、地点信息、饮酒日期和照片记录。

## 2. 产品结构

### 2.1 首页入口

首页包含三个独立入口：

| 入口 | 作用 |
| --- | --- |
| 选一杯 | 根据味道、香气、酒感和口感推荐一杯酒 |
| 识一杯 | 解析酒名、基酒、单种配料或多种配料组合 |
| 记一杯 | 手动新增、查看、编辑和删除饮酒记录 |

首页品牌文案：

> 杯中事  
> 今晚，喝点什么。

### 2.2 页面清单

| 页面 | 主要功能 |
| --- | --- |
| 首页 | 展示品牌和三个核心入口 |
| 偏好选择页 | 完成四类偏好选择并发起推荐 |
| 推荐结果页 | 展示单杯推荐、推荐原因及换杯操作 |
| 解析输入页 | 输入酒名或配料 |
| 解析结果页 | 展示酒款或材料组合的风味分析 |
| 酒单列表页 | 查看本地饮酒记录 |
| 记录编辑页 | 新增或编辑饮酒记录 |

### 2.3 页面流转

```text
首页
├─ 选一杯 → 偏好选择 → 推荐结果
│                      ├─ 换一杯 → 推荐结果
│                      └─ 返回重选 → 偏好选择
├─ 识一杯 → 解析输入 → 酒款解析结果 / 配料组合结果
│                         └─ 重新输入 → 解析输入
└─ 记一杯 → 酒单列表
           ├─ 新增记录 → 保存 → 酒单列表
           └─ 编辑记录 → 保存 / 删除 → 酒单列表
```

## 3. 功能需求

### 3.1 选一杯

#### 3.1.1 页面形式

- 所有偏好项位于同一个纵向滚动页面，不拆成分步流程。
- 页面标题为“今晚想喝点什么”。
- 底部固定主按钮“选一杯”。
- 未满足必选条件时，按钮不可用；点击或提交时应明确指出缺少的选择。

#### 3.1.2 推荐维度

| 维度 | 选项 | 规则 |
| --- | --- | --- |
| 味道 | 酸、甜、苦、咸、鲜、辣 | 至少选择一种，可多选；每种选择低、中、高 |
| 香气 | 花香、果香、草本、木质、香料、奶香、随机 | 必须单选 |
| 酒感 | 轻柔、适中、强烈、随机 | 必须单选 |
| 口感 | 清爽、顺滑、醇厚、随机 | 必须单选 |

补充规则：

- 未选中的味道表示“不作为本次推荐条件”，不表示排斥该味道。
- “随机”表示该维度不参与筛选，但仍代表用户主动完成了选择。
- 酒感为强制选择维度；用户不能在没有任何明确味道偏好的情况下发起推荐。

#### 3.1.3 推荐结果

每次只展示一个主推荐，内容包括：

- 中文酒名与英文名。
- 一句客观风味概述。
- 完整风味标签。
- 酸、甜、苦、咸、鲜、辣中与该酒相关的等级信息。
- 香气、酒感和口感。
- 主要配料名称，不展示毫升数、比例或制作步骤。
- “为什么适合你”：对应用户所选条件逐项解释匹配关系。
- 一段有生活气息的饮用感受。
- “换一杯”和“返回重选”两个操作。

文案原则：先清楚说明，再给感受。氛围文案应平实、克制、具体，不堆砌形容词，不仿写任何在世或已故作家的具体句式。

#### 3.1.4 换一杯

- 保留当前全部偏好条件，推荐另一杯匹配酒款。
- 当前会话内优先排除已经展示过的酒款。
- 连续换三次后显示轻提示：“已经换了几杯，要不要重新选选口味？”
- 提示操作为“继续换”和“重新选择”。提示不强制打断用户。
- 候选酒款耗尽时，可清空本轮已展示集合后继续推荐，并避免立即重复上一杯。

### 3.2 识一杯

#### 3.2.1 输入

- 使用一个多行输入框。
- 占位文案：“输入酒名、英文名或主要配料”。
- 支持中文逗号、英文逗号、顿号、分号、斜杠、空格和换行分隔多种配料。
- 输入可能包含酒名、英文名、别名、基酒和普通配料。

#### 3.2.2 识别优先级

1. 优先在内置数据库中匹配完整酒名、英文名和常见别名。
2. 只要输入中识别出明确酒名，优先按完整酒款输出解析结果。
3. 酒名之外的已识别材料可作为补充信息展示，但不改变酒款主体结论。
4. 未匹配到酒名时，再识别基酒和普通配料。
5. 识别到一种材料时，解析该材料本身及其在鸡尾酒中的常见风味作用。
6. 识别到多种材料时，根据组合规则给出整体味道、香气、酒感和口感判断。
7. 对无法识别的输入明确标注，不虚构分析结果。

#### 3.2.3 酒款解析结果

- 中文名、英文名和已收录别名。
- 客观风味描述。
- 酸甜苦咸鲜辣等级。
- 香气、酒感和口感。
- 主要配料名称。
- 一段有生活气息的饮用感受。

#### 3.2.4 材料组合解析结果

- 已识别材料。
- 未识别内容，如有。
- 组合后可能形成的整体味道。
- 酸甜苦咸鲜辣等级估计。
- 主要香气、酒感和口感。
- 各材料在组合中的主要作用。
- 一段简短、克制的整体感受。
- 明示说明：实际结果会受酒款比例、品牌和调制方式影响。

#### 3.2.5 第一版能力边界

- 解析仅基于内置酒款、材料数据和本地组合规则。
- 不调用生成式 AI，不保证覆盖酒吧自创酒款或罕见材料。
- 没有可靠结果时，应提示用户换一种写法或减少输入内容。

### 3.3 记一杯

#### 3.3.1 记录字段

| 字段 | 是否必填 | 说明 |
| --- | --- | --- |
| 酒名 | 条件必填 | 可单独填写；与配料至少填写一个 |
| 配料 | 条件必填 | 可填写一个或多个；与酒名至少填写一个 |
| 感受 | 选填 | 用户自由记录饮用感受 |

#### 3.3.2 功能

- 手动新增记录。
- 查看记录列表和记录详情。
- 编辑已有记录。
- 删除记录，删除前二次确认。
- 按最近编辑时间倒序排列。
- 第一版仅保存在当前设备本地。
- 推荐和解析结果页不提供“记下这杯”快捷入口。

#### 3.3.3 空状态

没有记录时展示简短空状态，并提供“记一杯”主操作，不使用社交或成就激励文案。

## 4. 数据与规则

### 4.1 酒款数据范围

第一版同时收录：

- 经典鸡尾酒。
- 国内酒吧酒单中常见的现代酒款。
- 常见基酒和高频配料。

建议首批收录 40 至 50 款含酒精鸡尾酒。具体清单在数据制作阶段评审，不作为本 PRD 已确认数量。

### 4.2 酒款核心字段

```js
{
  id: "gin-tonic",
  name: "金汤力",
  englishName: "Gin & Tonic",
  aliases: ["金汤力", "Gin Tonic", "G&T"],
  ingredients: ["金酒", "汤力水", "青柠"],
  taste: {
    sour: 1,
    sweet: 1,
    bitter: 2,
    salty: 0,
    umami: 0,
    spicy: 0
  },
  aroma: "草本",
  alcohol: "适中",
  texture: "清爽",
  tags: ["微苦", "柑橘", "草本", "清爽"],
  summary: "清爽微苦，带有柑橘和草本气息。",
  feeling: "第一口很凉，青柠把香气提起来，随后是汤力水干净的苦。",
  popularity: 80,
  enabled: true
}
```

说明：

- 内部味道等级建议统一为 0 至 3；0 表示不明显，1 至 3 对应低、中、高。
- 酒款可在内部保留多个香气标签，但面向用户推荐时选择一个主香气参与严格匹配，其余用于展示和次级评分。
- `popularity` 仅用于同分时排序和冷启动，不应长期固定推荐热门酒款。

### 4.3 推荐算法（MVP）

采用“加权匹配 + 必要过滤 + 去重扰动”的可解释规则：

1. 味道匹配为最高权重，计算用户已选择味道与酒款等级的距离。
2. 香气、酒感、口感分别参与匹配；选择“随机”时，该维度不计分也不扣分。
3. 对酒感差异设置较高惩罚，避免向希望轻柔的用户推荐烈酒感酒款。
4. 从最高分候选池中随机抽取一杯，候选池建议为前 3 至 5 名。
5. 排除当前会话已展示的酒款，直至候选池耗尽。
6. 推荐原因必须由实际命中的规则生成，不使用与选择无关的通用话术。

首版建议权重：

| 项目 | 建议权重 |
| --- | ---: |
| 味道匹配 | 50% |
| 酒感匹配 | 20% |
| 香气匹配 | 15% |
| 口感匹配 | 15% |

以上权重属于默认方案，应在酒款数据完成后通过典型案例测试调整。

### 4.4 本地数据

- 偏好选择只需保留在当前推荐会话中。
- 饮酒记录使用微信小程序本地存储。
- 每条记录包含唯一 ID、创建时间和更新时间，用于编辑、删除和排序。
- 本地数据丢失、换机不可同步等限制需在合适位置简洁说明。

## 5. 异常与状态

| 场景 | 处理方式 |
| --- | --- |
| 偏好未选完整 | 禁用提交，并在对应分组提示需要完成选择 |
| 推荐无严格匹配 | 放宽次要维度，明确提示“为你找了一杯最接近的” |
| 解析输入为空 | 不提交，提示输入酒名或配料 |
| 酒名未收录 | 尝试材料识别；仍无结果则提示未找到 |
| 部分材料未识别 | 展示已识别内容，同时列出未识别内容 |
| 记录酒名和配料均为空 | 阻止保存，提示至少填写其中一项 |
| 删除记录 | 二次确认后删除 |
| 本地存储失败 | 保留当前表单内容并提示稍后重试 |

## 6. 内容规范

每个酒款的内容分为两层：

1. 信息层：准确说明味道等级、主要香气、酒感、口感和配料。
2. 感受层：用具体的味觉、温度、气息和节奏描写饮用感受。

感受层要求：

- 平实、松弛、克制，避免故作深沉。
- 不使用“酸得舒展”等为了文艺而生造的表达。
- 不把抽象情绪代替真实风味信息。
- 不宣称酒精具有疗愈、减压或健康作用。
- 避免鼓励过量饮酒；在合适位置呈现理性饮酒提示。

## 7. 非功能需求

- 主要操作在常见手机尺寸下可单手完成。
- 输入、选择和返回操作不应丢失当前内容。
- 首屏和核心交互应快速响应；内置数据不依赖网络即可使用。
- 所有按钮具有清晰的按下、禁用和加载状态。
- 正文与背景满足基本可读性要求，不能以纸张纹理牺牲对比度。
- 交互目标建议不小于 44 × 44 px。

## 8. MVP 验收标准

### 8.1 选一杯

- 用户至少选择一种味道，并完成香气、酒感和口感选择后才可推荐。
- 每次只返回一杯已收录酒款。
- 结果与用户选择存在可解释的对应关系。
- 换一杯保留条件并避免立即重复。
- 第三次换杯后出现可忽略的重新选择提示。

### 8.2 识一杯

- 可识别已收录酒款的中文名、英文名和常见别名。
- 酒名与材料同时出现时，优先输出酒款解析。
- 多种材料可使用常见中英文标点、空格或换行分隔。
- 未匹配酒名时，可输出已识别材料组合的整体风味。
- 未识别内容被明确告知，不生成虚假结论。

### 8.3 记一杯

- 酒名或配料至少填写一项即可保存。
- 感受可以为空。
- 记录可新增、编辑、删除并在重启小程序后保留。

## 9. 后续版本候选

以下内容不进入第一版：

- 云端同步和跨设备记录。
- 从推荐或解析结果快捷生成记录。
- 拍照识别酒单和 AI 解析未收录酒款。
- 酒吧和地理位置信息。
- 图片、日期筛选、收藏、评分和分享。
- 无酒精鸡尾酒与家庭调酒教程。

## 10. 待验证项

以下事项不阻塞原型和开发，但应在数据及体验测试阶段确认：

- 首批酒款的最终数量和完整清单。
- 味道、酒感、香气和口感的具体算法权重。
- 酒款只有一个主香气用于选择匹配，是否足以覆盖真实体验。
- “鲜”和“咸”匹配酒款较少时，是否需要特殊引导或扩充现代酒款。
- 首轮真实用户是否能理解“木质”和“香料”；必要时在选项旁提供极短示例说明。
