# 杯中事技术设计文档

> 文档版本：V1.0  
> 对应产品版本：微信小程序 MVP  
> 需求基线：《杯中事 PRD》V1.0、《杯中事 UI 设计规范》V1.0  
> 更新日期：2026-07-17

## 1. 文档目的

本文档定义杯中事 MVP 的技术选型、系统架构、页面结构、数据模型、核心算法、本地存储、异常处理和开发约束，作为编码、联调、测试和后续维护的共同依据。

第一版没有账号、后端或生成式 AI。酒款推荐、酒名识别、配料组合分析和饮酒记录均在微信小程序客户端完成。

## 2. 技术目标

- 在无网络环境下完成选酒、解析已收录内容和饮酒记录管理。
- 推荐结果可解释、可复现测试，不依赖不可控的生成内容。
- 酒款和配料数据与页面代码解耦，便于持续扩充和校订。
- 核心规则写成无页面依赖的纯函数，支持单元测试。
- 第一版结构保持简单，同时为未来云同步和 AI 解析保留接口边界。

## 3. 默认技术选型

以下属于 MVP 默认技术决策。若正式开发前已有团队统一技术栈，可在不改变业务契约的前提下替换。

| 类别 | 选型 | 说明 |
| --- | --- | --- |
| 客户端 | 微信原生小程序 | 包体轻、依赖少，适合仅面向微信的第一版 |
| 语言 | TypeScript | 为酒款数据、规则函数和页面参数提供类型约束 |
| 视图 | WXML + WXSS | 使用原生组件和样式体系 |
| 状态管理 | 页面状态 + 轻量会话 Store | MVP 不引入大型状态管理库 |
| 数据源 | 本地 TypeScript/JSON 数据模块 | 酒款、配料、别名和文案随小程序发布 |
| 持久化 | `wx.setStorageSync` / `wx.getStorageSync` | 仅保存饮酒记录和必要的本地数据版本 |
| 测试 | Jest 或 Vitest + 微信开发者工具 + 真机 | 纯逻辑自动化，页面和兼容性人工/自动化验收 |
| 代码质量 | ESLint + Prettier + TypeScript 检查 | 保持数据与逻辑一致性 |

暂不引入：后端服务、云开发、数据库、登录、网络请求层、AI SDK、跨端框架和远程埋点平台。

## 4. 总体架构

```text
┌─────────────────────────────────────────┐
│                 页面层                   │
│ 首页 / 偏好 / 推荐 / 解析 / 酒单 / 编辑  │
└───────────────────┬─────────────────────┘
                    │ 调用
┌───────────────────▼─────────────────────┐
│               业务服务层                │
│ 推荐引擎 / 输入解析器 / 组合分析 / 记录服务 │
└──────────────┬────────────────┬─────────┘
               │                │
┌──────────────▼─────────┐  ┌───▼───────────────┐
│       内置数据层        │  │    本地存储适配层   │
│ 酒款 / 配料 / 别名 / 规则 │  │ 饮酒记录 / 数据版本 │
└────────────────────────┘  └───────────────────┘
```

### 4.1 分层职责

| 层级 | 职责 | 不应承担的职责 |
| --- | --- | --- |
| 页面层 | 交互、校验提示、页面跳转、结果展示 | 算法计算、直接拼接持久化数据 |
| 业务服务层 | 推荐、识别、组合分析、推荐理由生成 | 依赖 WXML 组件或页面实例 |
| 内置数据层 | 提供经过校订的结构化酒款和配料数据 | 保存用户状态 |
| 存储适配层 | 读写、校验、迁移饮酒记录 | 页面展示、业务推荐 |

### 4.2 未来扩展边界

- 若增加云同步，仅替换 `RecordRepository` 实现，页面不直接感知存储位置。
- 若增加 AI 解析，在本地匹配失败后由 `RecognitionService` 调用远程实现；本地结果仍优先。
- 若改用跨端框架，数据类型、算法函数和测试用例可继续复用。

## 6. 页面与路由

### 6.1 页面注册

建议 `app.json` 页面顺序：

```json
{
  "pages": [
    "pages/home/index",
    "pages/preference/index",
    "pages/recommendation/index",
    "pages/recognition-input/index",
    "pages/recognition-result/index",
    "pages/records/index",
    "pages/record-edit/index"
  ]
}
```

### 6.2 页面传参与状态归属

| 页面 | 输入 | 状态来源 | 输出/跳转 |
| --- | --- | --- | --- |
| 首页 | 无 | 无 | 三个入口 |
| 偏好选择 | 可选：上次会话 | 推荐会话 Store | 推荐结果 |
| 推荐结果 | 当前偏好 | 推荐会话 Store | 换杯或返回重选 |
| 解析输入 | 可选：上次输入 | 页面内存 | 解析结果 |
| 解析结果 | `resultId` 或会话结果 | 解析会话缓存 | 重新输入 |
| 酒单列表 | 无 | 本地记录仓库 | 新增或编辑 |
| 记录编辑 | 可选：`recordId` | 本地记录仓库 | 保存或删除后返回 |

推荐偏好和当前解析结果属于短期会话数据，不写入永久本地存储。页面返回时需保留；小程序进程被系统回收后允许重置。

不要将完整酒款或解析对象序列化到 URL。页面之间只传短 ID，或通过轻量会话 Store 读取。

## 7. 核心类型与数据模型

### 7.1 公共枚举

```ts
type TasteKey = 'sour' | 'sweet' | 'bitter' | 'salty' | 'umami' | 'spicy';
type TasteLevel = 0 | 1 | 2 | 3;

type Aroma = 'floral' | 'fruity' | 'herbal' | 'woody' | 'spiced' | 'creamy';
type AlcoholFeeling = 'soft' | 'medium' | 'strong';
type Texture = 'refreshing' | 'smooth' | 'rich';
type OptionalChoice<T> = T | 'random';
```

显示层负责将枚举映射为中文，不在算法中使用中文文案作为判断条件。

### 7.2 酒款模型

```ts
interface Cocktail {
  id: string;
  name: string;
  englishName: string;
  aliases: string[];
  ingredientIds: string[];
  taste: Record<TasteKey, TasteLevel>;
  primaryAroma: Aroma;
  aromas: Aroma[];
  alcoholFeeling: AlcoholFeeling;
  texture: Texture;
  tags: string[];
  summary: string;
  feeling: string;
  popularity: number;
  enabled: boolean;
}
```

数据约束：

- `id` 使用稳定、唯一的 ASCII slug，发布后不随显示名称变更。
- `aliases` 包含中文名、英文常见写法和高频简称；所有搜索索引在构建时归一化。
- `primaryAroma` 必须出现在 `aromas` 中。
- 六种味道字段必须完整，值只能为 0 至 3。
- `summary` 与 `feeling` 均为人工校订内容，第一版不在运行时生成。
- `enabled=false` 的酒款不参与推荐和识别，但可保留用于数据维护。

### 7.3 配料模型

```ts
type IngredientCategory =
  | 'base-spirit'
  | 'liqueur'
  | 'wine'
  | 'bitter'
  | 'juice'
  | 'syrup'
  | 'mixer'
  | 'dairy'
  | 'herb-spice'
  | 'other';

interface Ingredient {
  id: string;
  name: string;
  englishName?: string;
  aliases: string[];
  category: IngredientCategory;
  tasteContribution: Partial<Record<TasteKey, TasteLevel>>;
  aromas: Aroma[];
  alcoholStrength: 0 | 1 | 2 | 3;
  textureContributions: Texture[];
  roleDescription: string;
  standaloneSummary: string;
  enabled: boolean;
}
```

`tasteContribution` 表示常见影响，不是化学意义上的精确值。配料组合没有比例时，系统只能给区间式判断，结果页必须显示不确定性说明。

### 7.4 用户偏好模型

```ts
interface Preference {
  tastes: Partial<Record<TasteKey, Exclude<TasteLevel, 0>>>;
  aroma: OptionalChoice<Aroma> | null;
  alcoholFeeling: OptionalChoice<AlcoholFeeling> | null;
  texture: OptionalChoice<Texture> | null;
}
```

校验规则：

- `tastes` 至少包含一个键。
- 已选择味道的值只能为 1、2、3。
- 香气、酒感和口感都不能为 `null`；`random` 是合法的主动选择。

### 7.5 推荐结果模型

```ts
interface RecommendationResult {
  cocktailId: string;
  score: number;
  matchType: 'strict' | 'relaxed';
  reasons: string[];
}

interface RecommendationSession {
  preference: Preference;
  shownCocktailIds: string[];
  changeCount: number;
  current?: RecommendationResult;
}
```

`changeCount` 只统计用户点击“换一杯”的次数；初次推荐不计入。第三次点击换杯并取得新结果后触发轻提示。

### 7.6 解析结果模型

```ts
interface CocktailRecognitionResult {
  type: 'cocktail';
  cocktailId: string;
  matchedAlias: string;
  recognizedExtraIngredientIds: string[];
  unrecognizedTokens: string[];
}

interface IngredientRecognitionResult {
  type: 'ingredient-combination';
  recognizedIngredientIds: string[];
  unrecognizedTokens: string[];
  estimatedTaste: Record<TasteKey, TasteLevel>;
  aromas: Aroma[];
  alcoholFeeling: AlcoholFeeling | null;
  texture: Texture | null;
  summary: string;
  feeling: string;
}

type RecognitionResult = CocktailRecognitionResult | IngredientRecognitionResult;
```

### 7.7 饮酒记录模型

```ts
interface DrinkRecord {
  id: string;
  drinkName: string;
  ingredientsText: string;
  feeling: string;
  createdAt: number;
  updatedAt: number;
}
```

校验规则：

- 写入前对三个文本字段执行首尾空白清理。
- `drinkName` 与 `ingredientsText` 至少一个非空。
- 建议长度上限：酒名 80 字符、配料 500 字符、感受 2000 字符；UI 达到上限时阻止继续输入并明确提示。
- `createdAt` 在编辑时保持不变，`updatedAt` 每次保存后更新。

## 8. 推荐引擎设计

### 8.1 输入与输出

```ts
function recommend(
  preference: Preference,
  cocktails: Cocktail[],
  excludedIds: Set<string>,
  rng?: () => number
): RecommendationResult | null;
```

注入 `rng` 便于测试固定随机结果。生产环境默认使用 `Math.random`。

### 8.2 味道距离

只计算用户主动选择的味道：

```text
tasteDistance = Σ |用户等级 - 酒款等级| / 已选味道数量
tasteScore = 1 - tasteDistance / 3
```

未选择的味道不计分也不扣分。用户选择“酸：高”时，酒款酸度 3 得满分，2 次之，0 差距最大。

### 8.3 分类维度得分

- 香气相同得 1，否则得 0；用户选择随机时不参与总权重。
- 酒感相同得 1，相邻等级得 0.5，跨两级得 0。
- 口感相同得 1，否则得 0；用户选择随机时不参与总权重。

当某维度为随机时，从分母中移除该项权重，剩余权重重新归一化，避免随机项变相降低总分。

### 8.4 权重

默认权重：

| 维度 | 权重 |
| --- | ---: |
| 味道 | 0.50 |
| 酒感 | 0.20 |
| 香气 | 0.15 |
| 口感 | 0.15 |

权重集中定义在配置模块，不散落在页面代码中。酒款数据完成后，以测试矩阵校准。

### 8.5 严格匹配与放宽

严格候选建议满足：

- 总分不低于配置阈值，默认建议 `0.65`。
- 用户明确选择轻柔酒感时，强烈酒感酒款不进入严格候选。
- 至少一个已选味道的等级差不超过 1。

若严格候选为空：

1. 保留味道和酒感权重。
2. 允许香气或口感不一致。
3. 选择总分最高的前 3 至 5 款，并将结果标记为 `relaxed`。
4. 页面显示“为你找了一杯最接近的”。

### 8.6 候选抽取和去重

1. 过滤 `enabled=false` 及会话中已经展示的酒款。
2. 按得分降序排列。
3. 取前 5 名中与最高分差距不超过配置阈值的候选，默认建议 `0.10`。
4. 使用注入的随机函数抽取一杯，避免相同偏好总出现同一酒款。
5. 候选耗尽时清空已展示集合，但保留当前酒款 ID，避免立即重复上一杯。

### 8.7 推荐理由生成

推荐理由由命中的结构化规则生成，不能从 `feeling` 中截取：

```ts
interface ReasonFact {
  dimension: 'taste' | 'aroma' | 'alcohol' | 'texture';
  key: string;
  requested: string;
  actual: string;
  match: 'exact' | 'near' | 'relaxed';
}
```

规则示例：

- 精确味道：“你选了明显酸味，这杯的柑橘酸度也很突出。”
- 相近味道：“你想要一点甜，这杯的甜度较轻，主要用来收住酸味。”
- 酒感：“你选择了轻柔酒感，这杯入口不冲，酒精感藏得比较后。”
- 香气：“你选择了草本香，这杯以金酒和香草气息为主。”

最终展示 2 至 4 条事实，合并为一段自然语言。不得声称未命中的维度符合用户偏好。

## 9. 识别与组合分析设计

### 9.1 输入归一化

输入处理顺序：

1. Unicode 规范化和全角英数转半角。
2. 英文字母转小写。
3. 合并连续空白，清理首尾空白。
4. 保留原始输入供页面回显。

重要约束：不能先按空格拆分全部输入。英文酒名和配料可能包含空格，例如 `Old Fashioned`、`Dry Vermouth`。识别器必须先在完整归一化文本中进行最长别名匹配，再处理剩余文本。

### 9.2 别名索引

应用启动或数据模块初始化时构建索引：

```ts
interface AliasIndexEntry {
  normalizedAlias: string;
  entityType: 'cocktail' | 'ingredient';
  entityId: string;
  sourceAlias: string;
}
```

- 按别名长度降序匹配，优先识别更具体的名称。
- 中文别名可直接连续匹配；英文别名校验单词边界，避免短词误命中长单词。
- 同一别名发生实体冲突时，数据校验必须报错并要求人工指定优先级，运行时不猜测。

### 9.3 分隔符

在完成酒名最长匹配后，对剩余文本支持以下边界：

- 中文逗号 `，`
- 英文逗号 `,`
- 顿号 `、`
- 中英文分号 `；;`
- 中英文斜杠 `/／`
- 换行和制表符
- 连续空格，仅在不属于已知英文别名时作为边界

空 Token 和重复材料应去除，结果保留首次出现顺序。

### 9.4 酒名优先规则

1. 在完整输入中搜索所有酒款别名。
2. 选择最长命中；长度相同时优先完整边界命中。
3. 命中酒款后返回 `type='cocktail'`。
4. 剩余文本继续识别配料，作为补充材料和未识别内容展示，但不重新计算整杯酒风味。

例如：

- `威士忌酸 蜂蜜`：主体为威士忌酸，蜂蜜为额外识别材料。
- `Old Fashioned, orange`：主体为古典，橙为额外材料。
- `威士忌、青柠、蜂蜜`：未命中酒款，进入配料组合分析。

### 9.5 配料组合估算

由于用户不提供比例，组合分析只给近似等级：

```text
每个味道：取贡献最高的两个材料
estimated = round(maxContribution × 0.7 + secondContribution × 0.3)
```

- 只有一个材料贡献时直接使用该材料等级。
- 多个相同材料去重，不重复加权。
- 酒感以基酒/利口酒等含酒精材料的最高强度为基础；存在多个强酒精材料时可上调一级，但不超过强烈。
- 香气按材料出现顺序、类别重要性和贡献频次排序，展示 1 至 3 个。
- 口感按奶制品、糖浆、果汁、苏打等规则确定；规则冲突时返回最主要口感，无法可靠判断时为 `null` 并显示“难以判断”。

组合文案采用模板和人工维护的材料作用描述拼装，不生成超出数据事实的内容。

### 9.6 无结果和部分结果

- 一个实体都未识别：返回无结果状态，不输出风味等级。
- 部分识别：正常输出已识别组合，同时单独列出未识别 Token。
- 输入仅含标点或空白：视为空输入。
- 输入过长：限制为 500 字符，超出后阻止提交，避免低性能和异常匹配。

## 10. 本地记录存储

### 10.1 存储键

```ts
const STORAGE_KEYS = {
  RECORDS: 'beizhongshi.records.v1',
  DATA_SCHEMA_VERSION: 'beizhongshi.schema.version'
} as const;
```

键名必须带产品前缀和版本，避免与其他模块冲突。

### 10.2 仓库接口

```ts
interface RecordRepository {
  list(): DrinkRecord[];
  get(id: string): DrinkRecord | null;
  create(input: RecordInput): DrinkRecord;
  update(id: string, input: RecordInput): DrinkRecord;
  remove(id: string): void;
}
```

### 10.3 写入策略

- 每次新增、编辑和删除后写入完整记录数组；MVP 数据量较小，优先保证实现简单和一致性。
- 写入前完成字段校验，写入成功后再更新页面状态。
- 捕获配额、序列化和系统异常；失败时保留表单内容并提示稍后重试。
- 列表读取后按 `updatedAt` 降序排序，不依赖存储数组原始顺序。

### 10.4 数据校验与恢复

- 读取结果必须判断是否为数组，并逐条校验必要字段。
- 单条损坏数据跳过并记录开发日志，不让整个列表崩溃。
- 整体 JSON 无法读取时返回空数组，并提示本地记录读取失败；不得自动覆盖原存储。
- 数据迁移函数按 schema 版本顺序执行，为后续字段变化保留空间。

### 10.5 ID

MVP 可使用时间戳、随机数和设备内计数组合生成 ID。不得只使用毫秒时间戳，以免快速连续保存产生冲突。

## 11. 状态管理

### 11.1 推荐会话

推荐会话 Store 负责：

- 当前偏好。
- 已展示酒款 ID。
- 当前推荐结果。
- 换杯次数。

生命周期：进入偏好页时创建；从推荐页返回重选时保留；返回首页后可清除。切到后台再回来不主动清除。

### 11.2 页面表单

- 解析输入和记录编辑草稿保存在页面 `data` 中。
- 页面内提交期间设置 `submitting=true`，防止重复点击。
- 保存失败不清空表单。
- 编辑页根据 `recordId` 加载快照，保存时以仓库最新数据为准更新。

## 12. 错误处理与日志

### 12.1 用户可见错误

| 错误 | 用户处理 |
| --- | --- |
| 偏好未完成 | 在对应分组提示，不跳页 |
| 推荐数据为空 | 提示暂时无法推荐并返回重选 |
| 解析无结果 | 提示换一种写法或减少输入 |
| 部分材料未识别 | 展示已识别结果和未识别内容 |
| 记录校验失败 | 聚焦酒名或配料区域 |
| 存储失败 | 保留内容并提示重试 |

### 12.2 开发日志

- 开发环境记录数据校验失败、别名冲突、存储异常和未覆盖的规则分支。
- 生产环境不输出用户输入全文和饮酒记录内容到远程日志。
- 第一版没有远程埋点；如后续增加，需先更新隐私说明并只收集必要事件。

## 13. 性能与包体

- 40 至 50 款酒和常见配料可直接随包发布，不需要数据库或分页接口。
- 别名索引初始化一次并缓存，避免每次输入重新构建。
- 推荐排序数据量小，可同步计算；页面仍需避免重复提交。
- 纸张纹理和插画使用 WebP/PNG 并按实际显示尺寸压缩。
- 非首屏大图可按页面分包或延迟加载；MVP 是否分包以最终资源体积决定。
- 不把字体全字库直接打入主包。优先使用系统字体；如使用字体资源，应做子集化并核对授权。

性能目标：

| 场景 | 目标 |
| --- | ---: |
| 本地推荐计算 | 50 ms 内 |
| 本地酒名/配料识别 | 100 ms 内 |
| 记录列表读取与排序（500 条内） | 100 ms 内 |
| 换杯视觉反馈 | 300 ms 内完成 |

## 14. 安全与隐私

- 不请求位置、相册、相机、通讯录和用户资料权限。
- 不上传饮酒记录和解析输入。
- 不收集手机号、OpenID 或设备标识。
- 明确告知记录仅保存在当前设备，清理缓存或换机可能丢失。
- 酒精内容应避免诱导过量饮酒，并按微信平台届时适用的类目、内容和年龄要求准备发布材料。

## 15. 数据质量工具

开发期提供数据校验脚本或测试，至少检查：

- 酒款和配料 ID 唯一。
- 名称及归一化别名无冲突。
- 所有 `ingredientIds` 均存在且启用。
- 六种味道等级完整且在 0 至 3 范围。
- 主香气包含于香气数组。
- 枚举值合法。
- 必需文案非空，字符串长度不过长。
- `enabled=true` 酒款可被至少一种合理偏好推荐到。

内容数据变更必须经过数据校验和典型推荐矩阵回归，不能只做页面目测。

## 16. 构建与环境

建议环境：

| 环境 | 用途 | 数据 |
| --- | --- | --- |
| 开发版 | 本地开发、规则调试 | 可启用测试数据和调试日志 |
| 体验版 | 产品、设计、测试验收 | 与生产相同酒款数据 |
| 正式版 | 微信审核与上线 | 关闭调试日志 |

配置通过编译常量区分，不允许用页面里的硬编码判断环境。

建议脚本：

```json
{
  "scripts": {
    "typecheck": "tsc --noEmit",
    "lint": "eslint .",
    "test": "jest",
    "test:coverage": "jest --coverage",
    "validate:data": "jest tests/data --runInBand"
  }
}
```

实际命令根据项目脚手架调整，但应保留等价能力。

## 17. 开发顺序

1. 建立 TypeScript 类型、数据校验和首批测试夹具。
2. 实现推荐引擎及单元测试。
3. 实现输入归一化、别名索引、酒名优先和组合分析测试。
4. 实现记录仓库及本地存储异常处理。
5. 搭建全局样式、基础组件和首页。
6. 完成选一杯流程及第三次换杯提示。
7. 完成识一杯流程和结果状态。
8. 完成记一杯的新增、编辑、删除和空状态。
9. 导入正式酒款数据与视觉资源，执行回归和真机测试。

## 18. 技术验收标准

- TypeScript 类型检查、Lint 和全部自动化测试通过。
- 核心业务逻辑不依赖页面实例，可在测试环境独立调用。
- 所有酒款和配料数据通过结构、引用和别名冲突校验。
- 推荐正确处理随机维度、放宽匹配、会话去重和第三次换杯计数。
- 识别正确处理中文/英文酒名、多种分隔符、英文名空格和酒名优先。
- 本地记录在新增、编辑、删除、重启和写入失败场景下行为符合 PRD。
- 无网络时三项核心功能均可使用。
- 页面不申请 MVP 范围外权限，不上传用户记录。
