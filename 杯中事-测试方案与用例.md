# 杯中事测试方案与用例

> 文档版本：V1.0  
> 对应产品版本：微信小程序 MVP  
> 测试基线：《杯中事 PRD》V1.0、《杯中事 UI 设计规范》V1.0、《杯中事技术设计文档》V1.0  
> 更新日期：2026-07-17

## 1. 测试目标

验证杯中事 MVP 的三个核心流程在微信小程序中正确、稳定且易用：

- 用户能完成四类偏好选择，并得到一杯与条件可解释对应的酒。
- 用户输入酒名或配料时，系统按酒名优先规则给出可信结果。
- 用户能在本机新增、编辑、删除并持续保存饮酒记录。
- 页面在常见 iOS、安卓尺寸下无文字溢出、遮挡、错位和不可操作状态。
- 无网络、数据异常和本地存储失败时，应用不崩溃、不伪造结果、不丢失当前表单。

## 2. 测试范围

### 2.1 范围内

- 首页及三个入口。
- 偏好选择、推荐算法、推荐理由、换一杯及第三次提示。
- 中文酒名、英文酒名、别名、基酒、单配料和多配料组合识别。
- 多种分隔符、部分识别、无法识别和酒名优先。
- 饮酒记录新增、列表、编辑、删除和本地持久化。
- 数据结构、引用、别名冲突和文案完整性。
- UI 状态、移动端适配、基础可访问性、性能和离线能力。

### 2.2 范围外

- 登录、账号、云端同步和跨设备数据。
- AI 解析、拍照识别和图片上传。
- 社交、分享社区、评分和酒吧地点。
- 无酒精鸡尾酒。
- 调酒比例、制作教程和家庭调酒流程。

## 3. 测试策略

### 3.1 测试层次

| 层次 | 重点 | 建议方式 |
| --- | --- | --- |
| 数据校验 | ID、枚举、引用、别名、文案完整性 | 自动化测试 |
| 单元测试 | 推荐、归一化、识别、组合估算、记录校验 | Jest/Vitest |
| 组件测试 | 选择器、按钮、输入和提示状态 | 小程序测试能力或开发者工具 |
| 流程测试 | 选一杯、识一杯、记一杯端到端流程 | 开发者工具 + 真机 |
| 视觉测试 | 排版、纹理、状态、尺寸适配 | 截图对比 + 人工检查 |
| 兼容性测试 | iOS、安卓、不同微信版本 | 真机/远程设备 |
| 探索性测试 | 小白用户理解、异常输入和边界操作 | 人工测试 |

### 3.2 自动化优先级

必须自动化：

- 数据校验。
- 推荐得分、随机维度、放宽规则和去重。
- 输入归一化、最长别名匹配和酒名优先。
- 多配料组合估算。
- 饮酒记录校验、排序、增删改和存储异常。

优先人工或截图验收：

- 纸纹与文字可读性。
- 首页构图和品牌辨识。
- 键盘、安全区、长文本与字体放大。
- 真机动效、点击手感和微信胶囊避让。

### 3.3 自动化实施约定

- 业务测试直接调用 `services`、`storage` 和 `utils` 中的纯逻辑，不通过页面实例间接测试算法。
- 微信存储 API 使用内存适配器或 Mock；正常、读取损坏和写入异常分别构造。
- 随机推荐必须注入固定 RNG，候选集合测试与随机分布测试分开执行。
- 时间相关测试使用固定时钟，避免 `createdAt`、`updatedAt` 和排序用例偶发失败。
- 每个用例自行创建夹具和存储状态，不依赖其他用例执行顺序。
- 正式酒款数据的参数化测试与小型算法夹具分开，便于识别“代码错误”和“内容数据错误”。

建议测试结构：

```text
tests/
├─ data/
│  ├─ cocktails.spec.ts
│  └─ ingredients.spec.ts
├─ unit/
│  ├─ recommendation-service.spec.ts
│  ├─ normalize-input.spec.ts
│  ├─ recognition-service.spec.ts
│  ├─ combination-service.spec.ts
│  └─ record-repository.spec.ts
├─ fixtures/
│  ├─ cocktails.ts
│  └─ ingredients.ts
└─ helpers/
   ├─ fixed-rng.ts
   ├─ fixed-clock.ts
   └─ memory-storage.ts
```

## 4. 测试环境

### 4.1 软件环境

- 最新稳定版微信开发者工具。
- 项目支持的最低基础库版本，以及最新稳定基础库版本。
- 至少一个接近正式发布配置的体验版。
- 自动化测试运行于项目锁定的 Node.js 版本。

最低基础库版本需在项目初始化后根据实际使用 API 确认，并写入发布检查清单。

### 4.2 设备矩阵

| 类别 | 建议设备/视口 | 重点 |
| --- | --- | --- |
| iOS 常规屏 | 390 × 844 左右 | 安全区、键盘、字体 |
| iOS 较小屏 | 375 × 812 或更小 | 首屏入口、长文本 |
| 安卓常规屏 | 360 × 800 左右 | 字体回退、布局、输入法 |
| 安卓窄屏 | 320 至 360 px 宽 | 标签换行、按钮文字 |
| 字体放大 | 系统大字体 | 文本换行、按钮自适应 |

至少使用一台真实 iPhone 和一台真实安卓设备完成发布前验收。

### 4.3 网络与存储条件

- 正常网络。
- 飞行模式/断网。
- 本地存储为空。
- 本地存在 1、100、500 条记录。
- 本地数据部分损坏。
- 模拟存储配额不足或写入失败。

## 5. 测试数据

### 5.1 基础酒款夹具

自动化环境至少准备下列特征酒款，不要求使用这些显示名称：

- 高酸、低甜、清爽、轻柔。
- 低酸、高甜、顺滑、适中。
- 中苦、草本、强烈、醇厚。
- 果香、轻柔、清爽。
- 木质、强烈、顺滑。
- 香料、适中、醇厚。
- 咸或鲜明显的酒款，用于稀疏维度测试。

每个维度必须有可形成精确匹配、相邻匹配和无严格匹配的组合。

### 5.2 识别夹具

- 中文酒名：`尼格罗尼`。
- 英文含空格酒名：`Old Fashioned`。
- 英文符号别名：`G&T`。
- 单基酒：`威士忌`。
- 单普通配料：`青柠`。
- 多配料：`威士忌、青柠、蜂蜜`。
- 酒名加配料：`威士忌酸 蜂蜜`。
- 部分未知：`威士忌、月光糖浆`。
- 完全未知：`月光酒`。

正式数据上线前，需将全部中文名、英文名和别名加入参数化回归测试。

## 6. 缺陷等级

| 等级 | 定义 | 示例 |
| --- | --- | --- |
| P0 阻断 | 无法启动、数据大面积破坏、核心流程全部不可用 | 小程序白屏；所有记录被覆盖 |
| P1 严重 | 一个核心功能不可用或结果明显违背规则 | 酒名存在却按配料分析；无法保存记录 |
| P2 一般 | 局部功能错误，有绕行方式 | 某分隔符不识别；第三次提示时机错误 |
| P3 轻微 | 文案、视觉或低影响体验问题 | 分隔线偏差；非关键文案不一致 |

## 7. 准入与准出

### 7.1 测试准入

- 需求、UI 和技术文档版本明确。
- 核心页面可运行，正式或完整测试数据已导入。
- TypeScript 检查、Lint 和开发自测通过。
- 测试包可在开发者工具及至少一台真机安装。

### 7.2 发布准出

- P0、P1 缺陷为 0。
- P2 缺陷已修复，或有产品明确接受且不影响核心流程。
- 核心自动化用例 100% 通过。
- 全部 P0/P1 手工用例 100% 通过，整体用例通过率不低于 95%。
- iOS、安卓核心流程和离线流程通过。
- 数据校验无错误，推荐典型矩阵经产品验收。
- 没有明显文字溢出、遮挡、安全区冲突和权限越界。

## 13. 推荐质量验收矩阵

推荐算法不仅要“算得对”，还要让结果符合饮用直觉。正式酒款数据完成后，由产品、内容和测试共同维护不少于 20 组典型偏好。

每组记录：

| 字段 | 说明 |
| --- | --- |
| 偏好输入 | 已选味道等级、香气、酒感、口感 |
| 应进入候选 | 可接受的 3 至 8 款酒 |
| 不应进入候选 | 明显冲突的酒款 |
| 原因关键词 | 推荐原因应出现的事实 |
| 宽松策略 | 是否允许放宽香气或口感 |

必须包含：

- 高酸低甜、果香、轻柔、清爽。
- 中苦、草本、适中、顺滑。
- 低甜、木质、强烈、醇厚。
- 辣味、香料、随机酒感、顺滑。
- 咸味或鲜味明显的稀疏候选。
- 香气/口感/酒感分别为随机的组合。
- 没有严格匹配、必须放宽的组合。

验收时固定随机函数，确认候选集合；另以真实随机运行 100 次，检查结果只来自候选池且不是长期固定同一款。

## 14. 探索性与用户测试

建议邀请 5 至 8 名符合目标画像的用户，不提前解释产品规则，完成以下任务：

1. 想象今晚想喝偏酸、果香、酒感轻柔的一杯酒，完成推荐。
2. 在“换一杯”三次后决定继续或重新选择。
3. 输入一个熟悉或酒单上的酒名，说明自己从结果中理解到了什么。
4. 输入三种配料，判断是否理解结果只是大致估计。
5. 只记得配料、不知道酒名时，完成一条饮酒记录。

观察指标：

- 是否能理解低/中/高表达的是想要的味道程度。
- 是否能理解“随机”是主动放开条件。
- 是否理解木质和香料。
- 是否能找到推荐原因和主要配料。
- 是否误以为组合分析是精确配方结果。
- 是否理解记录只保存在本机。

核心任务建议成功率不低于 80%；同一处被 2 名及以上用户误解时，应进入产品或文案修订清单。
